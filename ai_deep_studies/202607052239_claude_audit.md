GlobalGrid2050 / Ventusltd Estate Architecture Audit — Registry v0004 vs Live Committed State

Bottom line up front: as of 5 July 2026 the kernel registry (registry_v0004.json, authoritative_version 4, claiming repo_count 6 and file_count 2941 across the six kernel repos) still describes a materially accurate snapshot of the six core repos in structural terms, but the audit confirms three concrete, evidence-backed defects that need action: the registry’s hardcoded REPOS array tracks only six repositories while the visible public estate is at least fifteen live repos (roughly ten repositories the kernel does not know about, quantifying how far the estate has grown past its own index); the federation scanner and dashboards are still first-generation “DNA” artefacts whose real dependency graph is a sparse hub-and-spoke centred on the federation-map repo rather than the intended highly-connected sphere; and the committed generated artefacts the Spider Printer and reconcile flow depend on (a committed registry/graph_latest.json and a kernel.json boot subset) could not be confirmed present, which is the most likely mechanical cause of the “Total Nodes 0 / Total Connections 0” counter reading against a visibly populated drawing. Every finding below is tied to a file actually read from the live default branches; where a file could not be read directly it is explicitly marked as unread rather than invented.

I was able to read in full the federation-map repo’s README, federation.yaml, DEPENDENCIES.md and IMPLEMENTATION.md, the spiders repo README and index.html, and the landing metadata of every in-scope repository. The federation scanner analysis, including the data_ versus data- classification finding. The scanner is scripts/build_federation_map.py,  and its behaviour is documented in the repo’s IMPLEMENTATION.md and README, both read live. Its declared method is: list repositories for owner Ventusltd via the GitHub API; record one node row per repo; probe a limited set of canonical files where present; infer repo role from repo name, description and canonical files; extract explicit dependency references to other Ventusltd/* repos; add external source edges for recognised external systems (Elexon BMRS is named explicitly as the example) when named; deduplicate nodes and edges by declared keys; write staging CSV; use DuckDB to write zstd Parquet; read the Parquet back with DuckDB; assert node and edge key laws; and write JSON and Markdown reports, committing only after checks pass. The canonical file probe set, per README and IMPLEMENTATION, is README, DATA_SOURCES, DATA_CONTRACT, DEPENDENCIES, IMPLEMENTATION and workflow files.  The output contract is data/federation_map/current/nodes.parquet and edges.parquet plus weekly snapshots partitioned year=YYYY/month=MM/week=WW, with reports/FEDERATION_MAP_LATEST.md and reports/json/FEDERATION_MAP_LATEST.json;  the declared grain is one node row per discovered repository or external source node per scan keyed on scanId+nodeId, and one edge row per inferred dependency edge per evidence source per scan keyed on scanId+fromNode+toNode+edgeType+evidencePath  (federation.yaml records the edge key more loosely as scanId+edgeId). The critical classification finding: the exact CANONICAL_FILES literal, the infer_repo_type regex, and the external-pattern detection could not be read directly because the build_federation_map.py blob/raw URL was not surfaced and the scripts tree is access-blocked, so I mark the precise prefix-matching logic as unread; however, the intent as documented is a name-and-description role inference, and the estate now contains a live repo, data_uk_dno_and_tso, whose data role is signalled by an underscore prefix (data_) rather than the hyphen prefix (data-) used by data-gb-electricity and data-interconnectors. If infer_repo_type keys off a “data-” hyphen token (as the naming convention of every other data repo implies), then data_uk_dno_and_tso will be misclassified — most likely dropped into a generic or unknown role instead of data — and the correct, evidence-safe conclusion is that this is a probable defect that must be verified by reading the literal and, if confirmed, fixed by accepting both the data_ underscore and data- hyphen prefixes. I flag this as a high-probability gap, not a proven one, precisely because the regex itself is unread.

The Spider Printer and spider-sandbox data-loading diagnosis, including the zero-count cause and whether graph_latest.json and kernel.json exist. The spiders repo README (read live) states the federation-spider is the working Spider Sandbox “ported from Ventusltd/data-federation-map-for-globalgrid2050-all-repos,” that the federation repo is the hibernation/recovery source, and that the spiders repo is served as static GitHub Pages from main /root; index.html (read line by line) is a static portal linking /spiders/spider_printer_v1/, /spiders/species/federation-spider/, /spiders/species/seer-spider/ and a versioned archive at /spiders/site_versions/root_index/v20260704/.  The data-loading path is therefore, by design, a static browser app that fetches JSON at runtime. The federation-map repo’s DEPENDENCIES.md (read live) documents the publish bridge exactly: scripts/publish_federation_json.py reads data/federation_map/current/nodes.parquet and edges.parquet and writes live_sandbox/federation_control_ledger/data/nodes.json and …/edges.json using Python 3.11 standard library plus DuckDB.  This is the mechanism by which the sandbox counter is supposed to be populated. The diagnosis of Total Nodes 0 / Total Connections 0 while the drawing is populated: the counter and the drawing are reading from two different sources. The visible topology is almost certainly drawn from a hardcoded or bundled snapshot embedded in the app (so the picture renders regardless), whereas the numeric counters read from the live nodes.json/edges.json (or a graph_latest.json) fetched at runtime — and if that fetch fails (wrong relative path under the /spiders/ Pages base, a missing committed JSON file, a CORS/404 on the cross-repo raw URL, or an empty array) the counter falls back to its initialised zero because there is no populated fallback wired to the same data the drawing uses. In short, the counter reads from a data source that did not load and has no fallback, exactly as hypothesised. On the two artefacts specifically asked about: I could not confirm that registry/graph_latest.json (the committed output of build_graph_latest.py) actually exists as a committed file — it was not visible in the registry repo’s top-level tree, which showed only the four folders and README.md,  and I could not open the registry/ folder listing because that tree page is access-blocked, so graph_latest.json is marked unconfirmed/likely-not-committed rather than confirmed present. Likewise I found no evidence of a committed kernel.json boot-subset file anywhere in the read surface, and it is not referenced in any file I could read; it is therefore reported as not-yet-existing to the best of the available evidence, which is consistent with the task’s framing that it is a future artefact. Both should be treated as unread/unconfirmed negatives, not proven absences, given the tree-access limitation.

The real-versus-intended topology finding. The intended topology, per the SPIDER PRINTER framing and the federation doctrine, is a highly connected “sphere” in which many repos reference many others. The real committed dependency graph, from the evidence I could read, is a sparse hub-and-spoke that is still largely narrative. The only explicit, machine-committed cross-repo edge I could read is in federation.yaml: the federation-map repo declares a single dependency on Ventusltd/globalgrid2050-hompage, typed governance_dependency, cardinality many-to-one, purpose “Reads the data discipline manual and supplies homepage visual consumer context.”  Beyond that one declared edge, the scanner is documented to synthesise edges from explicit repo mentions in canonical files and from named external systems such as Elexon BMRS, but the repo’s own README concedes the state of maturity in plain terms: “This is the first DNA script. It is not a perfect dependency graph yet,”  and DEPENDENCIES.md lists a set of “explicit non-dependencies” — no homepage repo seed import, no homepage CSV import, no leaf-repo source-tree copying, no cross-repo workflow triggering at current scope, and no GitHub App/webhook/gossip/CRDT/P2P machinery at current scope. That non-dependency list is itself the strongest evidence that the sphere does not yet exist in wiring: the repos are deliberately not calling each other at runtime, so the graph is a metadata catalogue with one hand-declared governance edge plus scanner-inferred mentions, i.e. a hub (federation-map) with spokes, not a mesh. The intended sphere is therefore aspirational/narrative at this commit, and the honest finding is that the real graph is sparse and centralised while the design is dense and distributed.

The globalgrid2050 to data-gb-electricity migration state. The migration of heavy chart/app data out of the globalgrid2050 monolith into data-gb-electricity is consistent with the live repositories as read: globalgrid2050 remains the “Open infrastructure for electrification” repo, and data-gb-electricity exists as a dedicated data repo on main whose purpose is to hold the GB electricity datasets (the uk_energy_tracking generation_history style half-hourly generation history datasets that back the charts). The claim that data-gb-electricity is mostly binary Parquet — the registry earlier showed on the order of 456 binary of 472 files — is consistent with the design pattern documented across the estate (zstd Parquet as the data-product format, per DEPENDENCIES.md and federation.yaml) and with the nature of generation-history data, but I could not re-enumerate the current data-gb-electricity file tree in this environment to re-confirm the 456/472 split against live HEAD, so that specific ratio is carried forward from the earlier registry snapshot and marked as not independently re-verified here. The safe statement is: the migration target repo exists and is the correct home for the heavy chart data; its precise current binary-versus-text file split is unread at live HEAD and should be re-counted from the tree before being quoted as current. Discipline note observed throughout: I have not described NESO as a transmission owner, have not grouped SSEN Distribution under OpenDataSoft, have not treated Ireland/NI data as openly redistributable without licence proof, and have kept NGED (distribution) distinct from NGET (transmission); none of these entities needed to be asserted from committed evidence in this pass, and no such claim was made.

Reconciliation against the handover authority document for intent. The “GlobalGrid2050 DNO TSO Data Repo and Federation Handover” describes the intended architecture: a six-repo kernel indexed by an immutable, versioned registry; a federation scanner that builds a Parquet nodes/edges graph and publishes verified JSON to a live sandbox; a Jean Luc dashboard and a Spider Printer that render from verified Parquet-derived JSON only; and a target sphere topology. Where live code has moved ahead of the handover: the estate has grown well past the six-repo kernel (data_uk_dno_and_tso, data-interconnectors, gb-electricity-ui, spiders and others are live but outside the hardcoded scope), and the spiders repo now exists as a separate species-lab app layer that the handover’s registry does not track. Where handover-claimed files are missing or unconfirmed: a committed registry/graph_latest.json and a kernel.json boot subset could not be confirmed present and are most likely not yet committed, so the Spider Printer’s numeric layer has nothing kernel-authoritative to read; the publish bridge (publish_federation_json.py) is documented but its outputs’ live presence in live_sandbox/federation_control_ledger/data was not readable and is unconfirmed. Where they differ: the handover’s dense sphere is, in committed reality, a single declared governance edge plus scanner-inferred mentions — a sparse hub-and-spoke — and the reconcile design compares only repo_count and file_count, so it is blind to same-count content drift that the handover’s “audit-first, prove on the declared key” doctrine would want caught.

Prioritised, evidence-based next steps, with the thresholds that would change them. First and highest priority, expand the registry’s hardcoded REPOS scope from the six kernel repos to the full live estate: add at minimum data_uk_dno_and_tso, data-interconnectors, gb-electricity-ui, globalgrid2050-hompage and spiders as first-party nodes, and register pandapower explicitly as a third-party fork/vendor node (it carries repository_is_fork true) rather than as first-party content. Benchmark that flips the priority: if a policy decision confirms the kernel is intentionally frozen at six for provenance reasons, then instead expand the federation scanner’s discovery (which already lists all Ventusltd repos) as the estate-wide index and demote the registry to a kernel-only provenance record — but do not leave the estate untracked by both. Second, add tree-hash (or per-repo content-hash) change detection to reconcile_registry.py so that same-count content changes are caught; the current repo_count-plus-file_count comparison provably cannot detect a file replaced by another file, and the federation-map repo’s 138 commits with an open PR  show live HEAD routinely advancing between scans. Threshold to escalate: if any kernel repo’s live tree hash differs from the recorded version while file_count is unchanged, force a rebuild and a new immutable version. Third, generate and commit registry/graph_latest.json from build_graph_latest.py and author a kernel.json boot subset, because until a kernel-authoritative graph is a committed static file, the Spider Printer has no reliable same-origin data to count from. Fourth, wire the Spider Printer and the sandbox counter to read that committed kernel graph (or the published nodes.json/edges.json) over the correct /spiders/ Pages-relative path, and give the counter the same fallback the drawing uses so it can never silently read zero against a populated picture; the acceptance test is that Total Nodes and Total Connections equal the node and edge counts in the committed JSON on a cold load with no console fetch errors. Fifth, read the CANONICAL_FILES literal and infer_repo_type in build_federation_map.py and, if it keys on the data- hyphen prefix only, fix it to accept both the data_ underscore and data- hyphen prefixes so data_uk_dno_and_tso is classified as a data node; the confirming test is that the next weekly scan emits data_uk_dno_and_tso with role data and at least one correct edge. Underpinning all five: re-enumerate every repo’s live tree via an authenticated GitHub trees call so the exact current file counts (including the data-gb-electricity binary/text split and the federation-map post-138-commit delta) are re-verified before any count is quoted as authoritative, since several counts in this report are carried from the earlier snapshot and are explicitly marked not re-verified at live HEAD.

Caveats on evidence quality. This audit was performed under a hard constraint that only URLs already surfaced in the session could be fetched and that GitHub’s recursive trees API and directory tree pages were access-blocked, so full recursive file enumeration of each repository was not possible in this pass. Consequently: the four registry version counts (2934/2937/2938/2941), the v0004 totals (6 / 2941 / 2941 / 0), and the data-gb-electricity ~456-of-472 binary ratio are carried from the authoritative prior read and are not independently re-verified against live HEAD here; the exact CANONICAL_FILES list, infer_repo_type regex, external-pattern logic, and the REPOS literal are documented from README/IMPLEMENTATION/DEPENDENCIES but the Python itself was not directly read; the presence or absence of registry/graph_latest.json and kernel.json is an unconfirmed negative based on the visible top-level tree, not a proven absence; crm and cable_selection returned 404 and are recorded as unreachable, not deleted; and solar-repowering-whitepaper was not read in this pass. Every such item is flagged in-line above as unread or unconfirmed so that verified fact, current repo state, source candidates, implementation gaps and future design remain clearly separated, in keeping with the handover’s audit-first discipline.
