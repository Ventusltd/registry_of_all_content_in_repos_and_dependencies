GLOBALGRID2050 FEDERATION — AUDIT-FIRST FILE REPORT AND KERNEL REGISTRY

Scope and method note. This document is an audit-first read of the five public GitHub repositories owned by Ventusltd (Ventus Ltd). It records what was reachable and verified against what is only claimed. Two access constraints materially shaped what could be read directly and must be stated up front so the reader can judge confidence. First, GitHub tree/subfolder HTML pages (the /tree/main/<path> listings) are blocked to automated access by GitHub’s robots policy, so subfolder file enumeration below the repository root relies on the root listings, on documents that name their own siblings (READMEs, anchor/dependency files, feature manifests inside ARCHITECTURE.md), and on GitHub Pages output, rather than on a live directory walk of every subfolder. Second, the automated fetcher only retrieves URLs that have already appeared in a prior search or fetch result, and it strips script tags from HTML; the raw.githubusercontent.com host and blob URLs for deep files were frequently rejected as not-yet-seen. A dedicated subagent was tasked specifically with retrieving the source of scripts/build_federation_map.py and the spider app JavaScript; it was unable to obtain either file’s source under the same restrictions and correctly declined to fabricate code. Consequently, every statement below about the internal line-by-line logic of build_federation_map.py, publish_federation_json.py, the port_csv_to_parquet pipeline internals, and the spider app’s JavaScript is marked as documented-but-not-source-verified, and is derived from the repositories’ own README/IMPLEMENTATION/DEPENDENCIES/DATA_CONTRACT prose plus the rendered Pages output. Anything I could not read is explicitly marked not-read/unreachable rather than guessed. Files verified by direct read are: all five repository root trees; federation.yaml, anchor_AI_MUST_READ.md, DEPENDENCIES.md, DATA_CONTRACT.md, IMPLEMENTATION.md (federation-map repo); ARCHITECTURE.md (globalgrid2050); and the rendered spider_printer_v1 Pages app. The narrative body follows the user’s plain-prose preference; the kernel registry at the end is deliberately a deterministic link-first index.

PART A — THE FEDERATION AND ITS FIVE REPOSITORIES

The whole estate. GlobalGrid2050 is an intended “federation” of energy-data and energy-application repositories under the Ventusltd GitHub org, described by Ventus Ltd as an open platform for documenting and analysing electrical energy systems under electrification.  The five repositories in scope play distinct roles. globalgrid2050 is the original monolith: a GitHub-Pages static site plus a very large mixed store of GeoJSON layers, REPD atlas versions, cost estimators, knowledge folders, Python ETL, and GridBot automation. data-gb-electricity is the new dedicated GB electricity time-series data layer (Parquet) that is being carved out of the monolith. data_uk_dno_and_tso is the declared operator/network-area data spine (operators, licence areas, MPAN distributor lookup, transmission-ownership bridge). data-federation-map-for-globalgrid2050-all-repos is the backend metadata “control ledger” that models the estate itself as nodes and edges. spiders is the newest repo, a “species lab” whose first inhabitant is the federation Spider viewer, deployed to Pages. A sixth repository, Ventusltd/globalgrid2050-hompage, is referenced as authoritative by three of the five in-scope repos (it holds docs/DATA_DISCIPLINE_MANUAL.md and is named as a governance dependency in federation.yaml) but was not in the list of five and was not read; it is registered here only as an external cross-reference, not audited.

Reconciliation against the handover authority document. Where handover-claimed files may be missing or differ: several files that the governing docs name as canonical entry points could not be confirmed present because subfolders are robots-blocked — specifically schemas/federation_manifest.schema.json, config/status_rules.json, docs/FEDERATION_CONTROL_LEDGER_ARCHITECTURE.md, scripts/render_internal_dashboard.py in the federation-map repo, and config/sources.json, schemas/*, scripts/lib/* in data_uk_dno_and_tso. The real federation topology versus the intended sphere. The intended design is a highly connected “sphere” in which every repo is a node, every dependency an edge, and every edge carries evidence (anchor_AI_MUST_READ.md and DATA_CONTRACT.md). The real dependency graph that is actually declared in-tree is far sparser and closer to a hub-and-spoke than a sphere. The only machine-readable federation manifest that exists among the five is federation.yaml in the federation-map repo, and it declares exactly one dependency edge — a governance dependency on globalgrid2050-hompage  (a repo outside the five). None of the other four in-scope repos was found to ship its own federation.yaml, which by the estate’s own rule (“No repo is considered fully federated until it has a local federation.yaml or equivalent machine-readable manifest”) means the other four are not yet fully federated by the doctrine’s own definition. The edges that genuinely exist as evidence in read files are: data-gb-electricity → globalgrid2050 (backfill clones the monolith’s CSV tree; a real, code-backed dependency) and data-gb-electricity → globalgrid2050-hompage (governance, via the Data Discipline Manual reference); federation-map → globalgrid2050-hompage (governance, via federation.yaml and README); federation-map → Elexon BMRS (an external-source edge the scanner is documented to create when Elexon is named, and Elexon is named in data-gb-electricity); data-gb-electricity → Elexon BMRS/FUELINST/FUELHH/system-prices (external source edges, real endpoints); spiders → federation-map (the federation-spider is a port of the federation-map’s Spider Sandbox, a declared lineage) and spiders → live_sandbox/federation_control_ledger data (intended consumption of the publish-bridge JSON, unconfirmed). data_uk_dno_and_tso currently sits almost isolated in the real graph: its README lists many intended consumers (atlas overlays, SLD sandbox, spiders, battery repos, IDNO/DNO comparisons) but no read file demonstrates a live inbound or outbound edge yet; it is aspirational in the topology, not wired. Conclusion: the “sphere” is presently narrative, not real. The genuine graph is a small hub-and-spoke centred on the federation-map ledger and the retiring monolith, with one true code edge (gb-electricity→monolith), a handful of governance edges to the un-audited homepage repo, and external-source edges to Elexon; the dense inter-repo web the doctrine describes does not yet exist in declared manifests.
